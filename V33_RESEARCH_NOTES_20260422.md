# 好吃家 v33 巴黎配色强化版（2026-04-22）

## 这次我重新研究后的结论

### 1) 为什么你会感觉“按钮没反应”
这次核查里，上一版最危险的问题不是单个按钮文案，而是**历史版本拼接式覆盖**：

- `assets/core.js` 里同名函数被重复定义了很多次；
- `bindDom / bindEvents / renderAll / persistState / exportAllData` 这类核心函数存在多版本叠加；
- 同时历史版本很多，`index.html`、`core.js`、`sw.js` 共用同路径，部署到 GitHub Pages 后很容易被旧缓存和旧 Service Worker 继续顶回旧资源。

这次 v33 的处理不是继续在污染文件上补丁叠补丁，而是：

- 以 **v31 clean core** 为干净基底；
- 只把 **v32 真正需要的双食品库 / 本地三重保存 / 照片塑形 / 导出 / GitHub 同步** 功能回灌进去；
- 把最终 `core.js` 清成**无重复函数定义**的单文件；
- 所有关键资源统一加了新的版本号，避免旧缓存继续干扰按钮逻辑。

### 2) 审美方向这次怎么改
你这次点得很明确：

- **不要全站深色背景**；
- 只是把高级配色用于关键功能和重点层级；
- 整体要像“巴黎奥运会那种浅底 + 高级点缀色”的气质。

所以这次改成：

- 全站浅底奶油色 / 米色；
- 重点功能用你给的绿、紫、蓝、棕、青、红去做高级点缀；
- 3D 舞台仍然保留层次感，但不再是一整块深色压住界面；
- 卡片、弹窗、按钮、营养环统一改成浅底玻璃感。

### 3) 3D 和营养环怎么处理
- 继续保留首页 3D 人体；
- 6 条主营养环继续围绕模型，位置外推，尽量不挡主体；
- 骨关节 / 跟骨 / 膝 / 骨密度等状态，继续优先抬高 **钙 / 维生素 D3 / 维生素 K1/K2**；
- 加了更稳的模型启动容错：即使 3D CDN 受阻，数据层和按钮也不会一起失效。

### 4) 照片塑形怎么强化
继续保留并强化成 6 维：

- 肩线
- 胸背
- 腰腹
- 臀腿
- 四肢
- 腿长

同时保留：

- 上传正面全身照；
- 自动轮廓估测；
- 再手动微调；
- 应用到 3D 模型。

### 5) 本地数据层怎么做
这次继续把本地数据放到优先级最高：

- localStorage
- IndexedDB
- 最近本地完整备份历史

保留并强化：

- 导入全部
- 导出全部
- 身体 JSON
- 身体 CSV
- 饮食 CSV
- 恢复最近本地备份

### 6) GitHub 云同步怎么做
默认仓库信息继续预填：

- owner: `kawhikun`
- repo: `haochijia`
- path: `data/haochijia-user.json`

路线仍然是：

- **先本地保存拉满**；
- GitHub 作为补充同步；
- 同步配置仍然保留在数据层面板里。

### 7) 食品库和翻译到底有没有做全
这次我直接把三个真实数据文件重新核查了一遍：

| 文件 | 条目数 | 中文名缺失 | 英文名缺失 | 原始名缺失 | 条码缺失 |
|---|---:|---:|---:|---:|---:|
| foods-cn.min.json | 36,793 | 0 | 0 | 0 | 0 |
| foods-global.part01.min.json | 59,704 | 0 | 0 | 0 | 15 |
| foods-global.part02.min.json | 13,413 | 0 | 0 | 0 | 0 |

所以这次不是“假装分库”，而是：

- **中文库 / 国际库分文件加载**；
- 显示可切 `中文 / EN / 原始`；
- 搜索同时吃中文名、英文名、原始名、品牌、条码；
- 绝大多数翻译字段已经是齐的，少量缺的是个别国际食品的条码，不是名称翻译。

## 这次 v33 具体做了什么

- 改成浅底巴黎配色，不再整站深色；
- 重新整理 `core.js`，去掉重复函数定义；
- 强化按钮绑定链和初始化容错；
- 给资源路径统一加版本号，降低旧缓存干扰；
- Service Worker 改成新缓存名并清旧缓存；
- 食品库继续分中文 / 国际，语言继续分中文 / EN / 原始；
- 营养建议区继续保留，并且在保存身体记录后立即刷新；
- 本地保存 / 导入 / 导出 / 最近备份 / GitHub 同步入口继续保留；
- Three.js 远程模块增加多 CDN 回退。
