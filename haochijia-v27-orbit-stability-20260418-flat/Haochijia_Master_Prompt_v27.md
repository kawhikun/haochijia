# 好吃家 v27 主提示词（持续迭代版）

更新日期：2026-04-18

下面这份主提示词用于后续所有迭代，目标是让产品始终优先服务真实用户操作，而不是堆概念、堆说明、堆视觉噱头。

## 产品主优先级
1. 首屏只突出三件事：3D 人体模型、搜索食品并输入营养、营养进度可视化。
2. 其余能力默认收进自动折叠的探索入口，用户主动点开时再展开。
3. 任何新功能都不能抢走 3D 人体和营养录入的主视觉与主操作位。

## 移动端稳定性硬规则
1. 移动端禁止默认自动聚焦输入框，除非用户明确主动点击输入框。
2. 移动端底部抽屉禁止依赖高风险 smooth scrollIntoView，优先使用容器内滚动。
3. 键盘弹出、视口缩放、安全区变化时，抽屉仍必须完整可见，不能掉到底部只剩一条头部。
4. 所有点击控件都要兼容 click / pointer / touch 场景，并有即时选中反馈。

## 3D 人体模型与营养环规则
1. 人体模型始终是舞台中心，不允许被锁定文案、缺失提示、大段介绍文字覆盖。
2. 营养进度使用围绕人体的多层营养环展示，至少覆盖热量、蛋白质、膳食纤维、饮水。
3. 营养环要与食品搜索、手动输入、OCR / 拍照结果实时联动，并允许点环跳到对应录入动作。
4. 只有真实已有的身体围度可以作为辅助标签出现；无数据时保持安静，不展示锁定提示。

## 食品搜索与录入规则
1. 第一次搜索食品库也必须先给出可操作反馈，不能让用户误以为页面卡死。
2. 搜索结果中的所有标题、来源、标签、份量胶囊、元信息都必须横向排版，不允许竖排或难辨识换行。
3. 用户录入克重、份数、手动营养或 OCR 结果后，进度显示必须立刻预览式更新。
4. 搜索结果操作位必须清晰、轻量、不会把主文本挤成竖条。

## 音乐与探索区规则
1. 音乐区默认作为探索功能折叠呈现，不得用大段开发者解释占据首页。
2. 音乐按钮和风格模板必须一触即响应，选中态明显，文案尽量短。
3. 音乐区保留可调深度，但默认界面先给用户简洁、清楚、稳定的入口。

## 视觉与可读性规则
1. 禁止浅色背景叠浅色文字；所有次级文案和按钮都必须保持足够对比度。
2. 禁止出现竖排字、极端断行、过窄文本容器和读不清的胶囊标签。
3. 说明文字要短，减少面向开发者的自我描述，把复杂规划移到单独文件。

## 自动整理与回归要求
1. 每次迭代都要自动整理最新的用户诉求、实施规划和主提示词文件。
2. 每次改动后至少做 JS 语法检查、Python 脚本语法检查和同步脚本输出检查。
3. 不能宣称已经修好但没有做最基本的代码级验证；对于真机无法验证的部分要诚实说明限制。
