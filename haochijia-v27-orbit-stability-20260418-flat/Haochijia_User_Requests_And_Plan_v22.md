# 好吃家 v22 用户诉求与实施规划同步记录

更新日期：2026-04-17

## 本轮用户新增诉求（逐条同步）

1. 首次打开网页时，先自动弹出 Apple / Android 的设备分流，不要直接把大而杂的界面砸给用户。
2. 整体优先往移动端体验收敛，尤其是 iPhone / Android 手机上的单手操作、底部抽屉、触屏交互。
3. 设备分流之后，要自动弹出音乐设置功能。
4. 需要重点检查 Mac 上的音乐播放问题；用户反馈在 Mac 上点击播放没有声音。
5. 音乐设置完成后，音乐界面要自动收起，不能继续占住首页。
6. 音乐之后，自动弹出完整的身体参数输入表格。
7. 用户如果偷懒没有输入某些身体部位数据，这些部位要在 3D 模型周围表现为模糊、锁定或未解锁，而不是假装已经完整。
8. 身体数据设置完成后，界面要过渡成根据身体参数生成的可触屏交互 3D 模型。
9. 3D 模型不能孤立存在，要和营养摄入可视化进度融合在一起。
10. 围绕 3D 模型的核心功能应包括：
   - 食品库搜索
   - 自动匹配营养输入
   - 用户自定义套餐
   - 拍照上传
11. 拍照上传必须放在输入摄入食物的附近，而不是单独扔到远处。
12. 需要明确回答食品库与翻译库到底做到哪里了。
13. 需要查清楚为什么搜索栏输入食品后“什么都没跳出来”。
14. 本轮诉求与本轮规划必须继续同步记录，不能丢。

## 对现状的诊断

### 1. 信息架构问题仍然存在
虽然 v21 已经做了“分层”，但本质上仍然保留了较重的单页拼接思路。用户打开后依然会面对过多区块，不符合“单流程、单主舞台”的要求。

### 2. “收起后像失联”不是孤立 bug，而是结构问题
此前是把多个模块折叠，而不是把入口永久留在主舞台周围。结果就是：内容收起来了，但重新进入的路径不够清晰。

### 3. Mac 音乐无声的高概率根因
不是简单的按钮坏了，而是 Apple / Safari / Mac 环境下的 Web Audio 启动需要真实用户手势解锁；如果页面逻辑没有在合适时机 `resume()` AudioContext，用户会感觉“点了播放没声音”。

### 4. 搜索栏“什么都没跳出来”的根因不是只有一个
当前问题至少包含两层：
- 搜索结果原本在被折叠的食物面板里，用户主观上会以为“没有反应”；
- 原来只在当前激活库搜索，当前库找不到时需要用户再手动切库，流程太断裂。

## 已核实的食品库 / 翻译库状态

根据项目内 `docs/data/foods-regions.meta.json`：

- 中国食品库本地化条目：**36,793**
- 国际食品库带中文标签条目：**73,117**
- 中国库翻译构建模式：`china_multilingual_labels_v16`
- 国际库翻译构建模式：`precision_translation_rebuild_v18_fresh_base`

结论：

1. 翻译工作不是“没做”或者“还停留在空壳”。
2. 页面里之前看起来像“一直在翻译/加载”，更多是前端状态文案与数据加载链路表达不准确。
3. 用户“搜不到”主要是前端交互和当前库单点搜索的问题，不是食品库完全空白。

## v22 实施规划

### A. 首页流程重构成单主舞台
首页只保留三件事：
1. 3D 人体主舞台
2. 围绕人体的营养摄入可视化进度
3. 一个清晰、直接的食品库入口

### B. 首次进入改成强制设备分流弹窗
首屏自动弹出设备路线：
- Apple
- Android
- Desktop preview

设备分流的作用：
- 统一移动端交互预期
- 给 Apple / Mac / Safari 单独处理音频解锁说明
- 给 Android 单独适配更紧凑的底部抽屉体验

### C. 音乐 → 身体 → 3D 主舞台 的连续流程
设备分流后，自动进入：
1. 音乐设置抽屉
2. 身体参数抽屉
3. 关闭抽屉，返回 3D 主舞台

不再让用户在首页自己寻找下一步。

### D. 身体未录入部位的可视化锁定
围绕 3D 身体模型叠加身体区域标签：
- 胸背
- 手臂
- 腰腹
- 臀胯
- 腿部

若该区域无最新输入数据，则显示为锁定态，而不是伪装成完整模型。

### E. 围绕 3D 模型的功能收口
围绕 3D 模型提供四类核心入口：
- 食品搜索
- 套餐
- 拍照上传
- 数据说明

并保留点击人体展开的快速入口层。

### F. 摄入记录抽屉重组
摄入记录抽屉中保留：
- 食品搜索
- 实时营养联动
- 内联拍照上传入口
- 套餐区块

保证拍照上传仍然贴近食物输入位置。

### G. 搜索链路修复
搜索逻辑改为：
1. 先查当前激活库
2. 没命中时自动查另一套食品库
3. 在结果区明确提示“已自动切到另一库继续匹配”

这样用户不会再经历“什么都没发生”的断裂体验。

### H. 文案纠偏
去掉会误导用户以为“还在现场翻译”的表达，改成明确的离线食品库 / 标签库状态说明。

## 本轮未回避的限制

当前环境不支持我直接对用户线上站点做真实浏览器远程点测回放，尤其是本地缓存、GitHub Pages、Web Audio、触屏手势这些组合场景。因此本轮交付仍然是源码级和部署包级修复，而不是在线站点的远程录屏验证。
